# Rig-A2A

Agent-to-Agent (A2A) protocol client integration for [Rig](https://github.com/0xPlaygrounds/rig).

`A2AClient` connects to any [A2A](https://a2a-protocol.org)-compliant agent by
fetching its well-known `AgentCard`, and each skill the card declares can be
attached to a Rig agent as a tool or called directly.

`rig-a2a` currently supports native targets. The upstream A2A client uses Tokio
networking, which is not available on `wasm32-unknown-unknown`.

## Installation

Add the companion crate to your `Cargo.toml`, along with the rig-core crate:

```toml
[dependencies]
rig-a2a = "0.40.0"
rig-core = "0.40.0"
```

You can also run `cargo add rig-a2a rig-core` to add the most recent versions
of the dependencies to your project.

The root `rig` facade also exposes this crate behind the `a2a` feature, at the
module path `rig::a2a`.

## Feature flags

| feature      | default | enables                                  |
|--------------|---------|------------------------------------------|
| `rustls`     | yes     | rustls TLS for the HTTP transport        |
| `native-tls` | no      | native-tls for the HTTP transport        |

The JSON-RPC and REST HTTP bindings are wired up. The client selects the first
supported interface in the order advertised by the remote `AgentCard`, and
copies that interface's optional `tenant` into each request as required by A2A.
Agent-card discovery and protocol requests have a five-minute default timeout.
Use `A2AClient::builder().timeout(...)` to override it.

## Usage

`A2AAgentBuilderExt::a2a_tools` folds every skill on the remote agent into a
Rig agent's tool set when `.build()` runs:

```rust,no_run
use rig_core::providers::openai;
use rig_core::client::{CompletionClient, ProviderClient};
use rig_core::completion::Prompt;
use rig_a2a::{A2AAgentBuilderExt, A2AClient};

# async fn run() -> anyhow::Result<()> {
let client = openai::Client::from_env()?;
let remote = A2AClient::from_url("http://localhost:8080").await?;

let agent = client
    .agent(openai::GPT_5_2)
    .a2a_tools(&remote)
    .build();

let reply = agent.prompt("Greet me.").await?;
println!("{reply}");
# Ok(()) }
```

See [`examples/a2a_client.rs`](./examples/a2a_client.rs) for a runnable
version pointed at a live A2A server.

## Conversation threading

Per the [A2A specification](https://a2a-protocol.org/latest/specification/),
`contextId` and `taskId` are generated by the server and echoed back by
clients — this crate never generates them. `A2AClient::message` sends a
single message; chain `.context(id)` (and `.task(id)` to resume a paused
task) with ids taken from a previous response:

```rust,no_run
use rig_a2a::SendMessageResponse;

# async fn run(client: rig_a2a::A2AClient) -> Result<(), rig_a2a::A2AError> {
let SendMessageResponse::Task(task) = client.message("turn 1").send().await? else {
    return Ok(());
};
client.message("turn 2").context(&task.context_id).send().await?;
# Ok(()) }
```

Tool calls are stateless unless the model echoes ids surfaced by a previous
tool response:

- A completed task appends `[A2A TASK Completed contextId=<id>]` to its
  output, so the model can continue the conversation by echoing the
  `contextId` argument.
- A direct message response appends `[A2A MESSAGE contextId=<id>]` for the
  same reason. If the server associated the message with a task, the marker
  also includes its server-generated `taskId` so the model can continue it.
- A task paused in `input-required` surfaces
  `[INPUT REQUIRED contextId=<id> taskId=<id>] prompt=<text>`; the model
  resumes it by echoing both ids with the requested input.
- Terminal failure states surface as
  `[A2A TASK <state> contextId=<id>] status=<text>`, without a `taskId` —
  the spec does not allow sending messages to tasks in a terminal state.

All remote-controlled strings in these markers are JSON-escaped.

## Agent card security

When building from a URL, `A2AClient` fetches the well-known `AgentCard` at
the origin root and rejects a selected interface whose URL points at a different
origin. This prevents a hostile card from redirecting A2A calls, along with any
default HTTP headers on your `reqwest::Client`, to an unrelated host. Trusted
deployments that intentionally publish cross-origin interfaces can opt in with
`A2AClient::builder().allow_cross_origin_interfaces(true)`.

HTTP redirects are disabled for both agent-card discovery and protocol calls,
including when supplying custom HTTP configuration through
`A2AClient::builder().http_client_builder(...)`.

Text projected from A2A responses is capped at 1 MiB.
